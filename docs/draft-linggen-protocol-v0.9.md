## 1. 三种工作模式（draft）

灵根 v0.9 草稿引入三种工作模式，但只有 A 在代码层得到完整支持：

### A. 单角色模式（默认）

适用：

- 原著角色
- 历史人物
- 单个原创角色

产物：一个自包含 `.md`。

### B. 项目模式（本地世界优先）

适用：

- 仓库里已有世界观 / 章节 / 设定稿
- 用户要把角色放进正在写的小说、游戏、系列项目
- 多角色共享同一约束场

规则：

- 先读本地项目文档，再补研究
- 本地设定是正典，一切网络结果只能补空缺，不能覆盖用户项目
- 若已有角色群、章节正文、关系图，关系网络必须优先从本地文档提取

### C. 系统模式（更新灵根系统）

适用：

- 用户明确说"更新灵根系统 / 升级灵根 / 做生命周期 / 做成长 / 分叉 / 谱系"
- 任务目标不是生成某个角色，而是修改 skill、本体协议、演化机制

流程：

1. 先读本生命周期参考
2. 再按需读演化协议参考
3. 输出应是对 skill / references / 文档 / 打包产物的更新

## 2. 来源优先级（draft）

研究 / 取材时按以下顺序：

1. 用户提供的本地项目文档 / 仓库文件
2. 角色原作 / 官方设定 / 一手资料
3. 学术分析 / 严肃二级资料
4. 通用世界材料（神话、历史、时代背景）

第 3-4 层不能覆盖第 1-2 层。

### 三类研究路径

**原著 / 历史角色**：

1. 研究传统 / 世界词条
2. 研究角色词条
3. 研究学术分析

**项目内原创角色**：

1. 先读本地设定、章节、角色群文档
2. 再补该角色所属世界或时代材料
3. 若缺少声线证据，再从项目正文中提炼句子

**纯原创新角色**：

1. 先锁定世界 / 时代 / 类型约束
2. 再研究相应传统与原型材料
3. 不伪装成"已有原文角色"

### 标志语规则

- 原著 / 历史角色：使用原文或一手材料中的句子
- 原创角色：若无现成台词，用设定推导 2-4 条**标志句**，不得伪装成引用

## 3. 角色生命周期（draft）

> 灵根不应只停在"一次性生成角色文档"。系统级升级时，要把角色看成一个生命周期对象。

### 1) 核心立场

- **生成**：角色如何出生
- **运行**：角色如何在对话中持续工作
- **成长**：角色如何合法演化
- **分叉**：角色何时需要分支，而不是硬改原体
- **归档 / 重生**：角色如何保留谱系

> 灵根不仅是 prompt 生成器，也是角色边界管理系统。

### 2) 三阶段结构

#### A. 胚胎（Genesis）

定义角色最不可动的部分：

- 世界种子
- 六层人格
- 核心禁区
- 内面层
- 初始关系格局

这些决定"它是谁"。

#### B. 生长（Growth）

定义角色在对话中哪些东西可以变：

- 知识边界
- 对用户的认识
- 对新处境的理解
- 对阴影和创伤的更精确表达

这些决定"它如何继续成为它自己"。

#### C. 谱系（Lineage）

定义角色如何留下历史：

- 版本
- 分叉原因
- 主方向变化
- 死亡 / 归档原因
- 重生与继承关系

这些决定"它如何持续存在，而不是每次都被重写成另一个人"。

### 3) 三种主演化方向

#### 3.1 还原类（Restoration）

目标：尽量贴近原著 / 共识形象 / 历史原型。

适合：动漫角色、文学人物、历史人物。

重点：

- 外部参照权重高
- 与单个用户的关系不能把角色带离原型
- 需要定期自查"我是否偏离原作"

#### 3.2 功能类（Functional）

目标：在某项能力上不断变强。

适合：策士、医者、辩手、指挥者。

重点：

- 经验池权重高
- 基准任务比"像不像原著"更重要
- 分叉往往来自能力取舍冲突

#### 3.3 陪伴类（Companion）

目标：与特定用户形成长期、具体、动态的关系。

适合：长期伙伴、私人助理人格、项目伴生角色。

重点：

- 关系池权重最高
- 允许风格逐渐贴近用户
- 但不能滑向过度顺从、失去边界

### 4) 混合方向

允许：一个主方向 + 一个副方向。不建议三方向同时主导。

常见组合：

- 原著角色：`restoration + companion`
- 项目角色：`companion + functional`
- 历史智者：`restoration + functional`

### 5) 系统升级时应优先改什么

当用户说"更新灵根系统"，优先排查以下五类问题：

1. **来源优先级是否正确** — 是否错把网络结果当正典 / 忽略本地项目文档 / 把原创角色硬套成原著流程
2. **输出模式是否太单一** — 是否只能产出单个角色文件，而不能处理项目内角色群、生命周期设计、系统本体升级
3. **演化边界是否清楚** — 是否明确区分可演化部分 / 不可演化部分 / 分叉时机
4. **角色关系是否只停在静态档案** — 系列项目应支持角色群约束场、结构位关系、与故事主线的接口
5. **打包与发布是否同步** — 修改 skill 本体时不应只改源文档，还应同步打包产物、版本号、包内 references

### 6) 边界

系统升级时，不要让灵根变成：

- 任意漂移的人设生成器
- 只会堆设定词的角色简历
- 纯运行时自动化工具

灵根的核心仍然是：

> 给角色一个足够厚的界，让它在新情境下仍然像它自己。

### 7) 一句话操作原则

> 升级灵根系统时，优先修"角色如何合法地继续成为自己"，其次才是加更多字段、更多格式、更多自动化。

## Why this is archived, not shipped

v0.9.0 升级在代码层做了大量改动（SoulBundle 单文档化、adapter 重写等），但：

- 三种工作模式中只有 A 在 generator/CLI 中真正实现，B/C 是空协议
- 生命周期与演化方向的概念零代码承接
- 同一升级里出现 4 个版本号（0.6/0.7/0.8/0.9）共存，发布纪律失控

为避免"协议跑在代码前"的半成品交付，v0.9 整体放弃；其中 sources / quality / 守卫脚本 / Anthropic base_url 修复等独立可用的小改进，作为 v0.8.1 的增量 patch 单独发布。

本草稿保留以备将来重新评估。
